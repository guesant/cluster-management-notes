> **Para quem é:** operadores de clusters K3s multinode que precisam de runbooks para cenários de falha.
> **Referência:** mais detalhes estão em [Falha e recuperação (blueprint)](../../guides/blueprints/k3s-multinode/failure-and-recovery/).

Clusters multinode K3s com etcd embarcado têm resiliência, mas também novos modos de falha. Este guia cobre os cenários mais comuns e como recuperar.

## Matriz de severidade

| Cenário | Nós vivos | Quorum OK? | Cluster funciona? | Downtime |
| --- | --- | --- | --- | --- |
| Perda de 1 agente em N agentes | 1 - (N-1) | Sim | ✅ | Minutos (reschedule) |
| Perda de 1 servidor em 3 | 2 | Sim | ✅ | Segundos (eleição) |
| Perda de 2 servidores em 3 | 1 | ❌ | ❌ | Horas ou perda total |
| Perda de quorum completo | &lt;50% | ❌ | ❌ | Perda total (sem backup) |

## Diagnóstico rápido

Quando o cluster para de responder:

```yaml
# 1. Verificar nós
sudo kubectl get nodes

# Se comando pendurar → quorum perdido

# 2. Tentar criar um pod (testa escrita)
kubectl run test --image=nginx
# Se "Unavailable" → quorum perdido

# 3. Verificar logs de etcd
sudo journalctl -u k3s -n 50 | grep etcd
# Se "apply request took too long" ou "dial tcp: connection refused" → nó isolado/down
```

## Cenário 1: Perda de 1 agente

**Sintomas:** pods em agente-1 desaparecem, ficam `Terminating` ou migram.

**Impacto:** workload downtime (minutos enquanto reschedula).

**Recuperação:**

```yaml
# 1. Aguardar 5 min (timeout kubelet)
sleep 300

# 2. Remover o nó
kubectl delete node agente-1

# 3. Tentar reparar o nó
ssh admin@agente-1
sudo systemctl restart k3s-agent

# 4. Verificar retorno
kubectl get nodes

# Ou, se não consegue reparar, remover permanentemente
# (já foi deletado do cluster acima)
```

## Cenário 2: Perda de 1 servidor

**Sintomas:** pods em servidor-0 desaparecem; lag na API (eleição de líder).

**Impacto:** segundos a ~30 segundos de instabilidade de API durante eleição.

**Recuperação:**

```yaml
# 1. Aguardar 30 segundos (eleição de novo líder)
sleep 30

# 2. Verificar que novo líder foi eleito
sudo kubectl get endpoints kubernetes

# 3. Reparar servidor-0 (se possível)
ssh admin@servidor-0
sudo systemctl restart k3s

# Se não consegue reparar, deixar fora por enquanto
# (cluster funciona com 2 servidores)
```

## Cenário 3: Perda de quorum (CRÍTICO)

**Sintomas:** `kubectl` commands ficar presos ou retornar "Unavailable"; não consegue criar/atualizar/deletar nada.

**Exemplo:** servidor-1 e servidor-2 caem; só servidor-0 vivo (quorum = 2, vivos = 1).

**Impacto:** cluster **totalmente indisponível** para escrita.

**Recuperação (com backup):**

```yaml
# 1. Parar todos os servidores
# Em cada servidor:
sudo systemctl stop k3s

# 2. Verificar que tem snapshot de etcd
ls -la /var/lib/rancher/k3s/server/db/snapshots/

# Esperado: arquivos .db

# 3. Restaurar em um dos servidores (ex.: servidor-0)
ssh admin@servidor-0
sudo k3s server \
  --cluster-reset \
  --cluster-reset-restore-path=/var/lib/rancher/k3s/server/db/snapshots/on-demand_2024-01-15T10_30_00Z.db

# K3s reinicia com etcd restaurado

# 4. Reparar os outros servidores
# Então reiniciar:
ssh admin@servidor-1 'sudo systemctl start k3s'
ssh admin@servidor-2 'sudo systemctl start k3s'

# 5. Verificar quorum restaurado
sudo kubectl get nodes

# Todos os 3 devem aparecer como Ready
```

**Recuperação (sem backup — perda total):**

```yaml
# Reconstruir cluster do zero (perda total de estado)
# Em cada servidor:
sudo systemctl stop k3s
sudo rm -rf /var/lib/rancher/k3s/

# Reinstalar servidor-0 como first-server
curl -sfL https://get.k3s.io | sh -s - server --cluster-init

# Readdicioná-los (mesma sequência de Fase 5)
```

## Cenário 4: Partição de rede (nós não conseguem se alcançar)

**Sintomas:** alguns nós desaparecem de `kubectl get nodes`; quorum pode quebrar.

**Exemplo:** roteador cai; servidor-0 isolado; servidor-1 e servidor-2 conseguem se falar.

**Impacto:**
- Servidor-0 fica sozinho, quorum quebrado **localmente** (nele)
- Servidor-1 + servidor-2 formam quorum = cluster funciona **naquela partição**

**Recuperação:**

```yaml
# 1. Reparar a rede (ligar roteador, consertar cabo, etc)

# 2. Aguardar 1-2 minutos (convergência de cluster)
sleep 120

# 3. Verificar que reconectaram
sudo kubectl get nodes

# Todos os 3 devem aparecer como Ready
```

**Cuidado:** se deixar a partição de rede por muito tempo, há chance de split-brain (servidor-0 criar estado diferente de servidor-1/2). Reconectar rápido evita isso.

## Cenário 5: Corrupção de etcd

**Sintomas:** `kubectl` commands falham com erros de corrupção; logs mostram "bad CRC" ou "inconsistent".

**Impacto:** cluster indisponível.

**Recuperação:**

```yaml
# 1. Parar todos os servidores
sudo systemctl stop k3s

# 2. Em um servidor, tentar reparar (se possível)
ssh admin@servidor-0
sudo k3s server \
  --cluster-reset-restore-path=/var/lib/rancher/k3s/server/db/snapshots/<snapshot-válido>

# Se não tem snapshot válido, perda total

# 3. Reiniciar cluster

sudo systemctl start k3s
ssh admin@servidor-1 'sudo systemctl start k3s'
ssh admin@servidor-2 'sudo systemctl start k3s'
```

## Checklist pré-falhas

Para estar preparado:

```yaml
☐ Backups de etcd configurados (snapshots automáticos)
  # No primeiro servidor:
  sudo ls -la /var/lib/rancher/k3s/server/db/snapshots/

☐ Snapshots testados (consegue restaurar?)
  # Testar restauração em ambiente de teste

☐ Monitoramento alertando sobre perda de nó
  # Prometheus + Alertmanager configurados

☐ Documentação de runbooks acessível (este documento!)

☐ Firewall entre nós verificado
  # Testar ping entre servidores

☐ Quorum é número ímpar ≥ 3
  # kubectl get nodes → servidores = 3, 5, 7...
```
