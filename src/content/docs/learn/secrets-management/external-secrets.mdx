> **Para quem é:** quem já decidiu usar um [secret store externo](../encryption-vs-secret-store/) e está avaliando se o External Secrets Operator é a integração certa.

O External Secrets Operator (ESO) é um projeto que sincroniza valores de múltiplos backends (Vault, OpenBao, AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, e outros) para Secrets Kubernetes, usando uma API declarativa comum — independente de qual backend está por trás.

## Como funciona

Um `SecretStore` (ou `ClusterSecretStore`) declara qual backend e credencial usar. Um `ExternalSecret` referencia um `SecretStore` e especifica quais chaves/caminhos buscar, gerando e mantendo atualizado um `Secret` Kubernetes correspondente.

```mermaid
flowchart LR
    accTitle: Fluxo de sincronização do External Secrets Operator
    accDescr: Um ExternalSecret referencia um SecretStore configurado com credencial de acesso; o operator busca o valor no backend externo e mantém o Secret Kubernetes sincronizado.

    ExternalSecret["ExternalSecret"] -->|"referencia"| SecretStore["SecretStore / ClusterSecretStore"]
    SecretStore -->|"credencial"| Backend["Backend externo (Vault, OpenBao, nuvem)"]
    Backend -->|"busca valor"| Operator["External Secrets Operator"]
    Operator --> Secret["Secret Kubernetes"]
```

Diferente do [Infisical Secrets Operator](../../../guides/tasks/secrets/install-infisical/), que é dedicado a um único backend (o próprio Infisical), o ESO é um projeto agnóstico de backend — a mesma API funciona com múltiplos provedores, o que facilita trocar de backend sem reescrever os manifestos de `ExternalSecret`.

## Alternativas

Um operator dedicado a um único backend (Infisical Secrets Operator, Vault Secrets Operator) tende a expor recursos mais específicos daquele backend, ao custo de acoplamento. O ESO troca especialização por portabilidade entre backends.

## Quando usar

Quando o ambiente já usa (ou pode vir a usar) mais de um backend de segredos, ou quando a portabilidade entre backends é uma prioridade explícita.

## Quando evitar

Se o ambiente já está comprometido com um único backend e não há expectativa de trocar, um operator dedicado a esse backend pode expor funcionalidades específicas que o ESO não cobre com a mesma profundidade.

## Decisões que isso implica

Como qualquer operator de sincronização de segredos, o ESO ainda enfrenta o [problema do bootstrap](../bootstrap-problem/) — a credencial usada pelo `SecretStore` para se autenticar no backend precisa ser aplicada manualmente ou via um método de autenticação nativo do Kubernetes.
