> **Público-alvo:** quem tem experiência com o [blueprint single-node](../k3s-single-node-gitops/) e quer escalar para múltiplos nós com alta disponibilidade.
> **Versões testadas:** K3s 1.36, Debian 12, topologia 3 servidores + N agentes.

Este blueprint estende o caminho single-node (Fase 2) para clusters K3s distribuídos em múltiplos nós. Os conceitos fundamentais — Gateway API, cert-manager, Argo CD, Longhorn — permanecem; o que muda é a topologia do control plane, a coordenação entre nós e os runbooks de operação/recuperação.

- [Requisitos de rede e portAS](network-requirements/) — comunicação entre nós, balanceamento da API, certificados TLS multi-SAN.
- [Inicializar e adicionar nós](first-server/) — bootstrap do primeiro servidor e registro de servidores/agentes adicionais.

## Diferenças do single-node

| Aspecto | Single-node | Multinode |
| --- | --- | --- |
| Control plane | 1 servidor | 3+ servidores (recomendado) ou 1 com datastore externo |
| Quorum | N/A | Número ímpar de servidores; perda de > 50% dos servidores = cluster travado |
| API endpoint | localhost ou IP do nó único | Load balancer ou DNS com múltiplos IPs |
| Workloads | Rodam sempre no nó único (sem redundância) | Distribuídos entre agentes; servidores podem ter workloads |
| Armazenamento | PVs ficam locais ao nó (sem replicação automática) | PVs replicados via Longhorn entre nós (se configurado) |
| Atualização | Reboot único | Rolling update: drain + update + uncordon nó a nó |
| Recuperação | Perda do nó = perda total | Perda de 1 agente = workloads rescheduled; perda de 1 servidor = cluster funciona se quorum > 50% |
