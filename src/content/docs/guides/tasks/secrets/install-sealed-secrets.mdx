> **Pré-requisitos:** kubeconfig com acesso administrativo à API, `kubeseal` instalado na estação administrativa.
> **Versões testadas:** Sealed Secrets 0.30 (Bitnami).

Sealed Secrets criptografa um `Secret` inteiro usando a chave pública de um controller que roda no cluster de destino. Só esse controller consegue decifrar o resultado — veja [SOPS ou Sealed Secrets](../../../../learn/secrets-management/sops-vs-sealed-secrets/) para a comparação com a alternativa baseada em SOPS.

## Instalar o controller

> **Executar em:** qualquer máquina com `KUBECONFIG`, Helm e acesso administrativo à API.

```yaml
read -r -p "Versão do chart sealed-secrets: " SEALED_SECRETS_VERSION

helm upgrade --install sealed-secrets sealed-secrets \
  --repo https://bitnami-labs.github.io/sealed-secrets \
  --version "${SEALED_SECRETS_VERSION}" \
  --namespace kube-system
```

## Criar um Secret selado

Com o controller instalado, use `kubeseal` para converter um `Secret` em claro (nunca commitado) em um `SealedSecret` (seguro para versionar):

> **Executar em:** estação administrativa com `KUBECONFIG` e `kubeseal`.

```yaml
read -r -p "Namespace do Secret: " SEALED_NAMESPACE
read -r -p "Nome do Secret: " SEALED_NAME
read -r -s -p "Valor da credencial: " SEALED_VALUE
printf '\n'

kubectl create secret generic "${SEALED_NAME}" \
  --namespace "${SEALED_NAMESPACE}" \
  --from-literal=value="${SEALED_VALUE}" \
  --dry-run=client -o yaml \
  | kubeseal --format yaml > "${SEALED_NAME}-sealed.yaml"
```

O arquivo `${SEALED_NAME}-sealed.yaml` resultante é seguro para commitar — só o controller com a chave privada correspondente consegue decifrá-lo.

## Aplicar o Secret selado

```yaml
kubectl apply -f "${SEALED_NAME}-sealed.yaml"
```

O controller detecta o `SealedSecret`, decifra e cria o `Secret` Kubernetes normal correspondente automaticamente.

## Validação

> **Executar em:** qualquer máquina com `KUBECONFIG` e acesso à API.

```yaml
kubectl --namespace "${SEALED_NAMESPACE}" get sealedsecret "${SEALED_NAME}"
kubectl --namespace "${SEALED_NAMESPACE}" get secret "${SEALED_NAME}"
```

Ambos devem existir; o `Secret` gerado é gerenciado automaticamente pelo controller a partir do `SealedSecret`.

## Troubleshooting

Se `kubeseal` falhar ao buscar o certificado público do controller, confirme conectividade com a API e que o controller está `Running` — `kubeseal` busca a chave pública diretamente do controller no momento da selagem.

## Rollback

```yaml
kubectl --namespace "${SEALED_NAMESPACE}" delete sealedsecret "${SEALED_NAME}"
```

Isso remove também o `Secret` gerado automaticamente pelo controller.

## Próximo passo

Faça backup do par de chaves do controller — sem ele, `SealedSecret`s existentes se tornam irrecuperáveis após uma reinstalação. Veja [estratégias de recuperação](../../../../learn/secrets-management/recovery-strategies/).
